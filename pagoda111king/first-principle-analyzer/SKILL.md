# first-principle-analyzer - 第一性原理分析器

**版本**：v0.1.0  
**定位**：底层思维引擎 - 用第一性原理分解复杂问题，提取基本真理，重构创新方案

---

## 📖 技能说明

第一性原理分析器是一个结构化的思维框架，强制用户完成「质疑 - 分解 - 验证 - 重构」的完整流程，并提供外部视角挑战隐含假设。

**核心价值**：
- 摆脱类比思维（「别人怎么做」）
- 回归基本真理（「物理上可能的是什么」）
- 重构创新方案（从基本真理演绎）

**阶段 6：报告生成**
输出完整的 Markdown 分析报告

## 🎯 实战案例

### 案例 1：技能系统设计优化（技术架构）

**背景**：
用户正在设计一个元技能系统，但不确定应该采用集中式还是去中心化架构。

**分析过程**：
1. **假设识别**：
   - 「技能系统必须有中央市场」
   - 「技能必须由人类手动创建」
   - 「技能发现必须通过搜索」

2. **逐层分解**：
   - 为什么技能系统必须有中央市场？→ 历史沿袭（npm/pip 模式）
   - 为什么技能必须由人类创建？→ 当前 AI 能力限制
   - 为什么技能发现必须通过搜索？→ 信息组织方式限制

3. **基本真理**：
   - 技能的本质：可复用的能力单元
   - 技能存在的目的：避免重复造轮子
   - 技能的核心属性：原子性、可组合性、可发现性

4. **重构方案**：
   - 方案 A：基于能力图谱的自动发现系统
   - 方案 B：AI 自动生成技能的系统
   - 方案 C：去中心化的技能网络

**输出结果**：
用户决定采用三层架构（增强层、编排层、进化层），并优先开发编排层技能。

**用户反馈**：「这个分析框架帮我跳出了「参考 npm 设计」的思维定式」

### 案例 2：是否应该创业（人生选择）

**背景**：
26 岁技术人员，有稳定工作但想创业，不确定是否应该辞职。

**分析过程**：
1. **假设识别**：
   - 「创业必须辞职全职投入」
   - 「26 岁是创业的最佳年龄」
   - 「必须先有完整产品才能开始」

2. **逐层分解**：
   - 为什么创业必须辞职？→ 传统观念，非必然
   - 为什么 26 岁是最佳年龄？→ 统计相关性，非因果
   - 为什么必须有完整产品？→ 资源限制思维

3. **基本真理**：
   - 创业的本质：验证价值假设
   - 成功的关键：快速迭代 + 用户反馈
   - 年龄的优势：学习能力强 + 试错成本低

4. **重构方案**：
   - 方案 A：副业验证，周末 MVP
   - 方案 B：内部创业，利用公司资源
   - 方案 C：先积累行业资源再创业

**输出结果**：
用户选择方案 A，开始副业开发技能系统，6 个月后验证可行再辞职。

**用户反馈**：「避免了冲动辞职，用最小成本验证了想法」

### 案例 3：新产品功能优先级（产品决策）

**背景**：
技能系统有 10 个待开发功能，资源有限，需要确定优先级。

**分析过程**：
1. **假设识别**：
   - 「用户想要的功能最重要」
   - 「技术难度低的功能应该先做」
   - 「竞品有的功能我们必须有」

2. **逐层分解**：
   - 为什么用户想要的功能最重要？→ 用户可能不知道自己要什么
   - 为什么技术难度低的先做？→ 短期思维，可能错过核心价值
   - 为什么竞品有的必须有？→ 跟随策略，非差异化

3. **基本真理**：
   - 产品成功的本质：解决用户未满足的核心需求
   - 资源分配原则：ROI 最大化
   - 差异化优势：独特价值主张

4. **重构方案**：
   - 方案 A：优先级 = (用户价值 × 差异化) / 开发成本
   - 方案 B：先做能验证核心假设的功能
   - 方案 C：聚焦单一场景做到极致

**输出结果**：
采用方案 A 公式，优先开发「技能编排」功能（高价值、高差异化、中等成本）。

**用户反馈**：「这个优先级公式成为产品决策的标准工具」

### 案例 4：技术栈选型（技术决策）

**背景**：
开发新技能系统，在 Node.js、Python、Rust 之间选择。

**分析过程**：
1. **假设识别**：
   - 「流行技术栈更好找工作」
   - 「性能是最重要的考量」
   - 「团队熟悉的技术最好」

2. **逐层分解**：
   - 为什么流行技术栈更好？→ 生态丰富，但可能过度
   - 为什么性能最重要？→ 对于技能系统，开发效率可能更重要
   - 为什么团队熟悉的技术最好？→ 短期效率 vs 长期维护

3. **基本真理**：
   - 技术栈的本质：实现产品目标的工具
   - 选型标准：匹配业务需求 + 团队能力 + 长期维护
   - 技能系统的特点：IO 密集型、快速迭代、易扩展

4. **重构方案**：
   - 方案 A：Node.js（生态丰富、快速开发）
   - 方案 B：Python（AI 生态好、语法简洁）
   - 方案 C：混合架构（核心 Rust + 业务 Node.js）

**输出结果**：
选择 Node.js，因为：生态丰富（npm 技能多）、团队熟悉、快速迭代。

**用户反馈**：「避免了「技术炫技」陷阱，选择了最务实的方案」

### 案例 5：个人 IP 定位（职业规划）

**背景**：
技术人员想建立个人 IP，不确定应该定位为什么方向。

**分析过程**：
1. **假设识别**：
   - 「个人 IP 必须垂直细分」
   - 「粉丝越多越好」
   - 「必须日更才能成功」

2. **逐层分解**：
   - 为什么必须垂直细分？→ 算法推荐逻辑，非用户价值
   - 为什么粉丝越多越好？→ 虚荣指标，非商业价值
   - 为什么必须日更？→ 平台规则，非内容质量

3. **基本真理**：
   - 个人 IP 的本质：信任资产积累
   - 商业价值 = 粉丝质量 × 转化率 × 客单价
   - 可持续的关键：内容质量 + 个人热情

4. **重构方案**：
   - 方案 A：技术 + 哲学交叉定位（稀缺生态位）
   - 方案 B：技能开发教学（市场需求大）
   - 方案 C：AI 应用实战（趋势红利）

**输出结果**：
选择方案 A，定位「技术哲学实践者」，差异化竞争。

**用户反馈**：「找到了既有热情又有市场空间的生态位」

## 🧠 核心思维模式

### Elon Musk 模式
> 「从物理学角度思考：什么是真正正确的？什么是可能的？然后从那里往上推理。」

**应用**：
- 质疑「一直如此」的假设
- 分解到物理/经济/逻辑的基本真理
- 从基本真理重新构建

### Linux Kernel 模式
> 「文档与代码同等重要，过程透明化降低协作成本。」

**应用**：
- 分析流程必须文档化
- 每个阶段有明确输出物
- 分层让不同用户快速找到所需

### 亚里士多德模式
> 「第一原理是一个基本的、不证自明的命题，不能被从任何其他命题中推导出来。」

**应用**：
- 定义什么是「基本真理」
- 验证机制确认已到达基本真理
- 重构必须是演绎推理，而非归纳
